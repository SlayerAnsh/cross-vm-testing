# Changelog

All notable changes to this project are documented here. The format follows Keep a Changelog, and the project adheres to Semantic Versioning.

## [Unreleased]

### Added (pipeline suites: dependency gated phases with world handoff)

* `cross-vm-config`: suite phases (`[[suite.<name>.phases]]`) express dependency gated phases within a suite. Each phase names a `profile`, optional `needs` (earlier phases that must have passed), and `world` (`fresh`, the default, builds a new environment; `inherit` continues from the exact `(Ctx, World)` a single named donor phase ended with). Structural rules are enforced at load time (a violation is a `ConfigError`, so `cross-vm validate` exits 3): `needs` may only name an earlier phase in the same suite, phase profiles are unique per suite, `world = "inherit"` requires exactly one `needs` entry whose both ends (donor and inheritor) build exactly one starting world (`invariant`, `endurance`, `scenario`, or `fuzz` with `cases == 1`), and a donor may feed at most one inheriting phase (fan out is a hard error for now). The legacy `profiles = [...]` list is normalized into dependency free, fresh world phases by the loader, so an existing config keeps behaving exactly as it did.
* `cross-vm-framework`: the CLI `run` selection path drives a suite's phases in declaration order, honoring `needs` gating (a phase whose dependency did not pass is skipped, contributing nothing to the exit code) and `world` (a passing donor phase stashes its ending `(Ctx, World)` in a per harness session slot, alive for the registry's lifetime, and the one inheriting phase takes it by move). An inheriting phase whose donor never stashed anything (it failed, or was already spent) fails loudly with `RunError::Invalid` rather than silently reusing stale state.
* `examples/cross-vm-tests`: a checked in `progressive` suite in `vault.cross-vm.toml` (`deposit-soak` then `mixed-after-deposits`, `world = "inherit"`) and a new `tests/cli_e2e.rs` test drive the pipeline end to end as a real subprocess.
* `docs/config-runs-spec.md`: new section 4.7 documents the phase schema and structural rules; section 6.5 documents the session slot semantics.
* Behavior note: shrink is forced off for an inherited phase's failure. A shrink rebuild always starts from a fresh setup, which cannot reproduce a donor handed state, so the raw (unshrunk) failing history is kept instead.
* Behavior note: a replay artifact written for an inherited phase's failure carries `world_source = "inherited"` in its `[replay]` provenance and warns in the log that a standalone `cross-vm replay` starts from a fresh setup and may not reproduce the same failure the pipeline run saw.

### Changed (dynamic operation weights)

* `cross-vm-framework`: operations now carry dynamic selection weights. `Harness::weight(ctx, world, kind)` (provided, default `1`) sets each kind's relative draw weight per draw, recomputed from the live world, so the mix shifts as state evolves; `0` excludes a kind while the state makes it meaningless. Every random mode (fuzz, invariant, endurance) draws through one weighted path: config supplied static weights (`KindMix::Weighted`) multiply the dynamic weight, `kinds` restricted and default pools use static weight `1`. A draw where every effective weight is `0` fails the run as `Infra`. `Harness::generate` is removed; kind biasing that previously overrode it now implements `weight` (see the counter, ping pong, and vault examples). Scenario, replay, and shrink runs use fixed operation lists and ignore weights.

### Changed (harness extraction: standalone `harness-core` / `harness-core-macros` crates)

* The property-testing harness split out of `cross-vm-framework` into two new, chain-agnostic crates: `harness-core` (`crates/harness`: the `Harness` trait, the mode-typed `Runner` and its fuzz/invariant/endurance/scenario drivers, the seedable `Prng`, per-op `Stats`, and the outcome types `Verdict`/`HarnessError`/`CheckOutcome`/`RunReport`/`Coverage`) and `harness-core-macros` (`crates/harness-macros`: `#[fuzz_runner]`/`#[invariant_runner]`/`#[endurance_runner]`). Neither crate depends on any chain type; `crates/harness/README.md` and `crates/harness/tests/pure_function.rs` show the whole crate driven with a plain `u8` counter and `Ctx = ()`, no `cross-vm-framework` in sight. `cross-vm-framework::harness` re-exports every `harness-core` type unchanged and adds only the two chain-shaped pieces: `Ctx` (a started `MultiChainEnv`) and `classify` (the response classifier). `cross-vm-macros` keeps `config_runner`; the three runner attribute macros left it for `harness-core-macros`, re-exported from `cross_vm_framework::prelude` as before.
* **`Harness::Ctx` is now a generic associated type**, `type Ctx;`, instead of the hardcoded cross-VM `Ctx`. Every trait method (`apply`, `check`, `advance`) is generic over it, so a harness with no live system under test (a pure function, a data structure) sets `type Ctx = ();` and never touches a chain. A cross-VM harness sets `type Ctx = cross_vm_framework::harness::Ctx;`, unchanged in shape.
* **`HarnessError` reshape.** `Infra` now carries `Box<dyn std::error::Error>` instead of the framework's `CrossVmError`, and `HarnessError` itself deliberately does **not** implement `std::error::Error` (a blanket `impl<E: std::error::Error + 'static> From<E> for HarnessError` replaces the old `From<CrossVmError>` / `From<EnvError>` impls, so any concrete error lifts via `?`; a reflexive `Error` impl on `HarnessError` would collide with that blanket `From<T> for T`). Two constructors are new: `HarnessError::infra(e)` and `HarnessError::bug(detail)`.
* **Silent behavior change, read this if you have a harness with endurance block progression.** `Harness::advance`'s default body changed from "advance every chain in the environment" (`ctx.advance_all(blocks).await`) to a plain no-op. Every in-repo harness that sets `EnduranceConfig::advance_blocks` already overrides `advance` explicitly with `ctx.advance_all(blocks).await`, so nothing in this workspace regressed, but an external cross-VM harness that relied on the old implicit default (set `advance_blocks` without overriding `advance`) will now silently stop advancing chain time or block height during an endurance run. Add the override back explicitly; there is no deprecation warning for this, since a generic default cannot know it is running against a chain-shaped `Ctx` at all.
* `EnduranceConfig` doc comments and coverage/report shapes are otherwise unchanged; a fixed seed's operation stream is still byte-identical across this extraction (verified by the golden-seed tests in `mechanics.rs`).

### Added

* `cross-vm-macros`: `CwExecuteFns` and `CwQueryFns` now support tuple (unnamed) enum variants. Unnamed fields become positional method args `arg0`, `arg1`, ... (cw-orch style), e.g. `ManageFactoryState(ManageFactoryState)` generates `manage_factory_state(wallet, arg0)`.
* `cross-vm-macros`: `CwExecuteFns` / `CwQueryFns` accept an enum-level `#[cross_vm(trait_name = "...")]` attribute that renames the generated trait (default `<EnumName>Fns`). This lets the same `ExecuteMsg` / `QueryMsg` also derive cw-orch's `ExecuteFns` / `QueryFns` without the two colliding on the `ExecuteMsgFns` / `QueryMsgFns` trait name.

The per-crate `[workspace]` tables, profiles, locks, `.cargo` configs, and Makefiles were removed. Path-dep consumers (`crates/cosmwasm`, `examples/common`) now point at `contracts/cosmwasm/contracts/<name>`, and the two `include_bytes!` sites read the optimized wasm from `contracts/cosmwasm/artifacts/counter.wasm`. `make compile-cosmwasm` runs one Docker workspace-optimizer pass (mounting the repo root so the counter/vault optional `cross-vm-*` path deps resolve, pinning the image toolchain over the repo `rust-toolchain.toml`) and relocates the emitted wasm into `contracts/cosmwasm/artifacts/`.

### Added (`#[config_runner]` macro bridging config to `cargo test`, spec P5)

* New `crates/framework/src/config/test_bridge.rs` (`cli`-gated): `run_profile_for_test(config_path, harness, setup, profile, case, expected_cases)` loads a `*.cross-vm.toml` via the P1 loader (`cross_vm_config::load`, a plain `std::env::var` closure, no `dotenvy`), resolves `profile`, and drives it as a `#[tokio::test]` body, panicking on any failure so a failing run fails the test. `case: Some(i)` drives only that fuzz case (asserting the freshly resolved profile's `cases` still equals `expected_cases`, the count the `#[config_runner]` macro read at compile time, panicking `"config changed since compile, rebuild"` on a mismatch); `case: None` drives the whole profile through a fresh, single-harness `Registry` (the invariant/scenario/endurance shape). Uses only the plain `Registry::register` bound, never `register_persistent`.
* `crates/framework/src/config/registry.rs`'s fuzz loop is factored: the setup/`Runner::fuzz`/`with_stats`/`run_with`/stats-collection sequence for one case is now `pub(super) run_one_fuzz_case`, called both by `run_profile`'s `Profile::Fuzz` arm (unchanged behavior, same `sub_seed`, same call order) and by `test_bridge::run_profile_for_test`. This is what keeps the fuzz golden stream identical whether a case is driven via `cross-vm run` or a `#[config_runner]`-generated `#[tokio::test]`; `parse_kind_selection`/`build_setup_request`/`resolve_base_seed`/`KindSelection` are now `pub(super)` so the bridge can reuse them without duplicating any run logic. Both fuzz golden-seed tests (`mechanics.rs`) and every existing registry test stay green unmodified.
* New `#[config_runner]` attribute macro (`crates/macros/src/config_runner.rs`, registered in `lib.rs`): `#[config_runner(config = "vault.cross-vm.toml", harness = VaultHarness, setup = vault_config_setup, profile = "smoke")] async fn name() {}`. At expansion time it reads `<CARGO_MANIFEST_DIR>/<config>` (a new `toml` dependency on the macros crate parses it) purely to learn the profile's `mode`/`cases`; a `fuzz` profile fans out one `#[tokio::test]` per case (`name_case_0` .. `name_case_<cases-1>`, mirroring `#[fuzz_runner]`'s fan-out, honoring a `[defaults].cases` fallback), any other mode emits a single `#[tokio::test] async fn name()`. Each generated test calls `cross_vm_framework::config::test_bridge::run_profile_for_test` with `concat!(env!("CARGO_MANIFEST_DIR"), "/", <config>)` (a stable absolute path regardless of the test binary's cwd) and passes the macro's own `cases` count as `expected_cases`, so a config edited without a rebuild fails loudly instead of silently under/over-running. Missing config/profile/mode/`cases`, or a non-`async` annotated fn, are `compile_error!`s with a clear message.
* `examples/integration-tests/tests/harness/config_runner.rs` (new, gated `#[cfg(feature = "fuzz")]`, wired into `tests/harness/main.rs`): `#[config_runner(config = "vault.cross-vm.toml", harness = VaultHarness, setup = vault_config_setup, profile = "smoke")]` fans out into `vault_smoke_config_case_0` .. `_case_7` (the `smoke` profile's `cases = 8`), passing deterministically on mocks. `cargo test -p cross-vm-integration-tests --features fuzz config_runner` runs all 8.

### Added (framework: `register_persistent` + `export_world`, spec P4 World-export portion)

* `Registry::register_persistent` (`crates/framework/src/config/registry.rs`): an opt-in registration variant identical to `Registry::register` plus one extra bound, `H::World: serde::Serialize`. A scenario profile registered this way may set `export_world = "path.json"`; after the run's `run_steps` completes (pass or fail — the state actually reached is what gets written), the final `World` is recovered via `Runner::into_parts` and serialized as pretty JSON to that path, creating missing parent directories and logging the written path at `info`. Both `register` and `register_persistent` now funnel into a private `register_inner`, which takes an `Option<WorldExportFn<H::World>>` (`register` passes `None`, `register_persistent` builds `Some(export_world_json::<H::World>)`); `WorldExportFn<W>`'s own type (`Rc<dyn Fn(&W, &Path) -> Result<(), RunError>>`) never mentions `Serialize`, so `register_inner`'s signature — and therefore `register`'s — stays free of the bound. `Harness` itself is unchanged.
* A profile that sets `export_world` against a harness registered with plain `register` (no exporter) is rejected twice: `Registry::validate` fails it up front via a new `validate_export_world` check (`RunError::Validation`, so `cross-vm validate` catches the mismatch offline), and the scenario run arm rejects it again at the top, before touching a chain, with `RunError::Invalid` (since `cross-vm run` never requires a prior `validate` call).
* New `RunError::Export(String)` variant: an `export_world` write failure (parent directory creation, JSON serialization, or the file write itself) after a successful, `register_persistent`-backed run. Not a discovered SUT bug, so it maps to the same infra exit code (`2`) as `Setup`/`Serialize` in `cli.rs`'s `exit_code_for_run_error`.
* `Cli::register_persistent` (`crates/framework/src/cli.rs`) mirrors `Cli::register`, delegating to the new registry method with the same added `H::World: Serialize` bound.
* Framework unit tests (`config::registry::tests`): `register_persistent_scenario_run_exports_final_world_as_json` and `..._exports_even_on_failure` reuse the existing in-module `MockHarness` (`World = u32`, already `Serialize`) to assert the exported file's contents match the exact final World value on both a passing and a failing run; `plain_register_scenario_with_export_world_fails_validate` and `..._fails_run` cover the non-persistent rejection at both `validate` and `run` time, including that a rejected run never writes the export file.

### Added (framework: replay artifacts, configurable shrink, `replay` subcommand, spec P4 replay-loop portion)

* `Runner::shrink_with_limit(failing, check_every, limit, rebuild)` (`crates/framework/src/harness/runner.rs`): the shrink budget is now a parameter instead of the hardcoded `DEFAULT_SHRINK_LIMIT` constant. `shrink_inner`/`try_candidate` thread `limit` through; the exhaustion log line now reports the actual limit in effect, not always the default. `shrink_with` is unchanged as public sugar over `shrink_with_limit(.., DEFAULT_SHRINK_LIMIT, ..)`, so every existing caller (the golden shrink tests in `mechanics.rs`, `run_and_shrink_with`) compiles and passes unmodified.
* Shrink on failure in the registry run closure (`crates/framework/src/config/registry.rs`): a new `maybe_shrink` helper runs after the fuzz/invariant/endurance driver returns a failing `RunReport`, whenever `resolved.shrink` is true (the profile's own `shrink` key, mode default true for fuzz/invariant and false for endurance per the schema). It rebuilds a fresh `(Ctx, World)` from the harness's own `setup` fn for each replay attempt (the failing fuzz case's own sub seed, or the profile's resolved seed for invariant/endurance), drives a scenario mode `Runner::shrink_with_limit`, and replaces the failure's `history` with the minimized sequence while keeping `step`/`op`/`kind` from the original failure untouched. Scenario profiles never shrink (concrete steps are not generative). `erase_report` (`crates/framework/src/config/erased.rs`) gained a `shrunk: bool` parameter that the caller now controls explicitly, replacing the old hardcoded `false`.
* New `config::artifact` module (`crates/framework/src/config/artifact.rs`): `write_replay_artifact(dir, source, resolved, report) -> io::Result<PathBuf>` builds a self contained, valid config document, one `[harness]`, the resolved `[[chain]]` declarations (kind/target/rpc_url and every per kind field, the RESOLVED rpc_url only, never a mnemonic or key), `[env]`, a `[replay]` provenance block (source profile, source mode, failure summary, `shrunk`, `framework_version`; ignored by the run schema, tolerated by the existing loader), and one `[profile.replay]` scenario profile holding the (possibly shrunk) op history as ordered steps. Serialization tries `toml::to_string_pretty` first; on any error (an amount over `i64::MAX`, TOML's signed 64 bit integer ceiling, while still fitting the `u64` range `serde_json::to_value` already required) it falls back to a sibling `*.replay.json` written with `serde_json::to_string_pretty`, identical structure. The written path is `<harness>-<profile>-<seed>-<unix-seconds>.replay.toml` (or `.replay.json`). `build_chain.rs` gained `spec_id_to_str`, the reverse of `parse_spec_id`, for writing a chain's hardfork back out as its canonical short name.
* CLI wiring (`crates/framework/src/cli.rs`): `run_selected` writes a replay artifact whenever a fuzz/invariant/endurance profile's report carries a failure (never for a scenario run or a pass), logging the written path and a "reproduce with: cross-vm replay <path>" hint at `info`; a write failure is logged at `warn` and never changes the run's own exit code. A new `Replay(ReplayArgs)` `Command` variant makes `cross-vm replay <artifact>` sugar for `run <artifact> --profile replay`, reusing `dispatch_run` verbatim, so an artifact's `.toml`/`.json` extension, the registry, and the exit code contract are all already correct with zero new machinery.
* `examples/integration-tests/src/boom.rs`: a tiny, deterministically failing harness (`Noop` always passes, `Boom` always fails the same way) registered alongside `vault` in the `cross-vm` bin, plus `boom.cross-vm.toml`. The vault harness has no reachable bug, so this gives `tests/cli_e2e.rs` a real subprocess level failure to write an artifact for, shrink, and replay, without touching the vault's own (deliberately correct) contract logic. New e2e tests: a failing fuzz profile writes a `.replay.toml` artifact whose history is minimized to the one op that matters and which itself passes `cross-vm validate`; `cross-vm replay <artifact>` reproduces the exact same failure (exit code 1); a passing profile writes no artifact at all.
* Framework unit tests: `config::registry::tests` gained a shrink integration suite over the existing in module `MockHarness` (Boom fails the same way regardless of preceding Pings, so a minimized history is deterministically length 1 independent of the seeded rng stream), covering `shrink = true` minimizing and marking `shrunk`, `shrink = false` leaving the raw history alone, a tiny `shrink_limit` still returning a reproducing sequence, and endurance's `shrink` mode default of false (with an explicit override test). `config::artifact::tests` covers the TOML round trip through `cross_vm_config::load`, the u128/i64 boundary triggering the JSON sidecar (asserting the sidecar's own bytes hold the value losslessly; see the note below), a passing run producing an empty step list, and the `Bug`/`Invariant` failure summary formatting. `cli::tests` covers artifact writing on failure, no artifact on a pass, and a full `run` then `dispatch_replay` round trip within the process (no subprocess needed for this one).
* Known limitation, out of this task's scope: `cross_vm_config::load`'s JSON path (`crates/config/src/lib.rs`'s `json_to_toml_value`) funnels every input format through an internal `toml::Value`, which has no integer representation above `i64::MAX`; a `.replay.json` sidecar's own bytes hold a `u64` range amount exactly (verified directly against the file), but a *second* hop back through `cross_vm_config::load` would itself downgrade that value to a float before it ever reaches a harness's `Operation::deserialize`. Values that fit `i64` (the documented, common case) round trip through either the TOML artifact or the JSON sidecar without issue.

### Added (framework: JSON run reports, `--json-report` wired, spec P4 JSON-report portion)

* `cross-vm-framework`'s `serde` feature now adds `Serialize` (never `Deserialize` — reports are write-only) to `Coverage`, `InvCoverage`, `FailureKind` (`crates/framework/src/harness/outcome.rs`) and `Stats`/`OpStat` (`crates/framework/src/harness/stats.rs`), each behind `#[cfg_attr(feature = "serde", derive(serde::Serialize))]` so the derives are fully inert without the feature: `cargo build -p cross-vm-framework --no-default-features --features cw,evm,solana,tron` still pulls in none of `cross-vm-framework`'s own `serde`/`serde_json` deps (confirmed via `cargo tree --depth 1`; the many transitive `serde` entries deeper in the tree come from the VM provider crates' own unrelated deps, e.g. `cosmwasm-std`, `alloy`). `Coverage` is `#[serde(transparent)]` over its inner `BTreeMap<String, InvCoverage>`, so it serializes as a bare JSON object keyed by invariant name (`{"balances_never_negative": {"held": 2, "skipped": 0, "violated": 0}}`), not wrapped in a newtype layer. `FailureKind` uses serde's default externally tagged representation. `OpStat` uses a hand-written `Serialize` impl rather than a derive, so its wire shape is `{"count", "accepted", "rejected", "bug", "infra", "total_ns", "mean_ns", "min_ns", "max_ns", "stddev_ns", "errors"}`: consumer-meaningful nanosecond timing derived via `OpStat::total()`/`avg()`/`min()`/`max()`/`stddev()`, never the raw `sum_sq_ns` variance accumulator under its internal name. `Stats` derives plain (non-transparent) `Serialize`, nesting as `{"ops": {"Deposit": {...}, ...}}`.
* `config::erased::{ErasedReport, ErasedFailure}` (cli-gated, so `serde` is always present) now derive `Serialize` directly (no `cfg_attr` needed). `ErasedReport::elapsed` and `ErasedFailure`'s fields serialize with `Duration`'s default serde shape (`{"secs": .., "nanos": ..}`); no custom millisecond `serialize_with` was added, favoring one predictable `Duration` shape across the whole report over shaving a nesting level.
* New `config::report` module (`crates/framework/src/config/report.rs`): `JsonReport { schema_version: u32, invocation: Invocation, profiles: &[ErasedReport] }` and `Invocation { config: &str, profiles: &[String], overrides: serde_json::Value }`, matching spec section 9's envelope exactly (`schema_version` is always `1`). `write_json_report(path, config, profiles, reports, overrides)` serializes with `serde_json::to_string_pretty` and writes to `path`, creating parent directories as needed.
* `cli.rs`'s `run_selected` no longer has a `TODO(T11)` stub: it accumulates every selected profile's `ErasedReport` into one `Vec` across the whole invocation (never writing per-profile), resolves the JSON report path with the CLI `--json-report` flag winning over a resolved profile's own `json_report` key (first one found, since the envelope goes to exactly one file), and calls `write_json_report` exactly once after the loop ends (whether it ran every selected profile or stopped early on `stop_on_failure`). `invocation.profiles` in the envelope is always the invocation's full selected-name list, not just the profiles that finished running. A new `overrides_json(&RunOptions)` builds the small `invocation.overrides` object from only the CLI-set run-shape scalars (`seed`/`ops`/`cases`/`duration_secs`/`target`/`target_chain`/`stats`/`check_every`/`no_shrink`) — deliberately never `json_report`/`artifacts_dir` (file paths, not run-shape knobs) and never any config value, so the envelope can never leak a config secret through this field. A `write_json_report` I/O failure (bad directory, permissions) folds into the invocation's combined exit code as `3` (the usage/config bucket, alongside an unresolvable profile name or malformed config) rather than `2` (reserved for infra failures discovered *during* a run), and is logged at `error`; a successful write logs the path at `info`.
* `examples/integration-tests/tests/cli_e2e.rs` gained `run_with_json_report_writes_a_schema_version_one_envelope`, driving the real `cross-vm` binary with `--json-report <tmp path>` and asserting `schema_version == 1` plus the written profile's stable fields (`profile`/`harness`/`mode`/`steps`; deliberately not the fuzz report's own per-case `seed`, which is a sub-seed derived from `--seed`, not the fixed value itself).

### Added (framework: endurance op-count/infra-tolerance/heartbeat/stop, ctrl-c wiring, spec P3 endurance portion)

* `cross-vm-framework`: `EnduranceConfig` gains four fields, each with a builder: `max_ops: Option<usize>` (`.max_ops(n)`, stop on op count OR `duration`, whichever hits first; default `None`), `max_consecutive_infra: usize` (`.max_consecutive_infra(n)`, tolerate up to `n` consecutive `FailureKind::Infra` failures before the run fails, resetting on the next success; default `0` preserves today's "fail on the first `Infra`" behavior), `heartbeat: Duration` (`.heartbeat(d)`, a periodic `info`-level progress log with steps/elapsed/coverage/stats; default 60s, `ZERO` disables it), and `stop: Option<Arc<AtomicBool>>` (`.stop(flag)`, a cooperative cancellation flag polled only at the top of the loop, never around an in-flight `apply`; default `None`). **Breaking**: `EnduranceConfig` is now `#[non_exhaustive]` — construct it with `EnduranceConfig::new(duration)` plus builder methods, never a struct literal (pre-1.0; every in-repo construction already used the builder pattern, so nothing in this repo needed updating).

A tolerated `Infra` still counts as a step and still runs the block-advance/delay for that iteration, so pacing is unaffected by tolerance.
* The registry's endurance `RUN` arm (`crates/framework/src/config/registry.rs`) is filled in: one `setup`, `EnduranceProfile` mapped onto `EnduranceConfig` (`--ops`/`RunOptions.ops` overrides `max_ops`, `--duration`/`RunOptions.duration` overrides `duration`; when a profile sets `max_ops` but no `duration`, a 50-year "effectively unbounded" fallback — not `Duration::MAX`, which overflows `Instant` arithmetic — lets `max_ops` alone govern, matching the loader's duration-or-`max_ops` validation), `opts.stop` wired to `.stop(..)` when present, driven with `run_with`, erased to an `ErasedReport` with `mode = "endurance"`. `RunError::UnsupportedMode` is no longer constructed by any mode (every mode the schema knows about is now implemented) but stays on the enum as a forward-compat exit for a future mode.
* `RunOptions` (`crates/framework/src/config/resolve.rs`) gains `stop: Option<Arc<AtomicBool>>`, the CLI-to-registry channel for the ctrl-c flag; `resolve_profile` never reads it (consumed by the registry's endurance run arm), matching `ops`/`cases`/`duration`'s existing pass-through pattern.

### Added (framework: scenario driver with per-step expectations, spec P3 scenario portion)

### Added (harness diagnostics: per-invariant coverage, opt-in per-op stats, sequence shrinking)

* `cross-vm-framework`: `RunReport` now carries a `coverage` field, a per-invariant tally (`held` / `skipped` / `violated`) keyed by the invariant's `Debug` name. It is collected on every run at no configuration cost and seeded from `Harness::invariants()` at run start, so an invariant that never fires (for example one silently skipped on one VM's path) still shows up with an all-zero total. `Coverage::uncovered()` lists those never-ran invariants and `log_summary` prints a coverage line flagging them. The existing aggregate `skipped` count now equals `coverage.total_skipped()`.
* `cross-vm-framework`: opt-in per-op diagnostics via `Runner::with_stats()`. When enabled, a run records (per op variant name) the outcome counts (accepted, rejected, bug, infra), `apply` wall-clock timing (min, avg, max, stddev), and an error breakdown grouped by reason. A summary prints at run end through the same `log_summary` seam, and `Runner::stats()` returns the collected `Stats`. Off by default, so the zero-config path collects nothing. Use it to catch the "80% of generated swaps revert, so the run tested almost nothing" failure mode.
* `cross-vm-framework`: `Runner<H, Scenario>::shrink(failing, rebuild)`, a generic greedy delta-debug minimizer that reduces a failing operation sequence to a near-minimal one that still fails the same way (same `FailureKind` discriminant, and for an invariant failure the same invariant name, so shrinking never converges on a different bug). It re-drives the runner on a fresh `(Ctx, World)` from the async `rebuild` closure for each attempt (a chunked pass, then a one-by-one pass). `run_and_shrink(ops, rebuild)` runs a sequence and, on failure, puts the minimized history in the returned report.

  ```rust
  let min = runner.shrink(report.failure.unwrap().history, || async {
      vault_setup(seed).await.expect("setup")
  }).await;
  ```
* `cross-vm-framework`: documented that transition-style invariants (comparing state before vs after one op) belong in `apply` (async, holds `Ctx`), stashing the snapshot in `World`, rather than in the sync `ContractBase` hooks (which hold no chain handle and cannot query state). The vault harness example gains a `DepositTransition` invariant that snapshots on-chain collateral before a deposit and diffs the live post-state.

### Changed (runner internals factored; no behavior change)

* `cross-vm-framework`: the runner's three drivers now share one machinery layer. A private `ReportBuilder` is the single `RunReport` construction path (replacing six hand-built literals, and keeping `skipped == coverage.total_skipped()` and `Failure::step == steps` in one place); an `OpSource` enum (generated vs fixed ops) folds the Fuzz/Invariant driver and the Scenario `run_fixed` into one shared `drive` loop; and the endurance final invariant sweep and `step`'s per-op check loop are one `sweep` helper. Endurance keeps its own wall-clock loop. Verified behavior-preserving three ways: a new golden-seed test pins the exact op sequence `seed = 42` generates (the rng draw order is a compatibility surface; recorded seeds must keep reproducing), the whole harness suite passes untouched, and the `RUST_LOG` debug output of a full run is byte-identical before and after (timestamps aside). `ReportBuilder::fail` is also the natural interception point for a future failure-corpus feature. One intended nit: the endurance final sweep now emits the same per-invariant `debug` lines the mid-run checks always had.

### Changed (harness diagnostics hardening)

* `cross-vm-framework`: `Stats` are now strictly per-run. Every driver resets the collected stats at entry, and `shrink` / `run_and_shrink` park them for the duration of the shrink (candidate replays neither tally into the caller's data nor print per-op summary blocks), so a report's stats always describe exactly the run that produced it. Previously stats accumulated across every run and shrink replay on the same runner.
* `cross-vm-framework`: `shrink` compares `Bug` failures by detail string, not just discriminant, so minimizing one bug can no longer converge on a *different* bug that also surfaces as `FailureKind::Bug`. Emit stable, state-independent bug messages from `apply` to benefit. Invariant failures still compare by invariant name and `Infra` failures by discriminant only (transport errors are noisy).
* `cross-vm-framework`: `shrink_with(failing, check_every, rebuild)` and `run_and_shrink_with(ops, check_every, rebuild)` replay candidates under the invariant-check cadence the original failure surfaced with, so a cadence-dependent invariant failure is not re-judged under per-op checking. The existing `shrink` / `run_and_shrink` delegate with `check_every = 1` (unchanged behavior).
* `cross-vm-framework`: shrink replay attempts are capped at `DEFAULT_SHRINK_LIMIT` (256, the analog of Foundry's `shrink_run_limit`); an irreducible sequence costs O(n log n) replays in ddmin's worst case, and on exhaustion the best sequence found so far is returned (with a `warn` log) instead of replaying unbounded.
* `cross-vm-framework`: `Runner::run(ops, Some(&[]), ..)` now returns an `Infra` failure report ("empty op-kind slice") instead of panicking inside the rng.
* `cross-vm-framework`: `Stats` timing accumulators are saturating and fully `u128` (`min`/`max` previously truncated through `u64` casts, and a pathological duration could overflow the sum-of-squares and abort the run); `op_label` bounds its fallback label for `Debug` renderings that start with a non-identifier character.

### Added (native value calls, shared mock clock, timestamp-aware block advance)

On a live chain the signer must already hold the value.

### Changed (endurance: RPC-safe block advance, base + jitter inter-op delay)

* `cross-vm-framework`: `EnduranceConfig` gains a `base_delay` knob (builder `base_delay(d)`). ### Added (cross-VM ping-pong relayer test)

* `cross-vm-integration-tests`: a cross-VM ping-pong test (`tests/cross_vm/ping_pong.rs`) that relays IBC-style packets between heterogeneous chains. A `PingPong` cross-VM wrapper (`tests/support/ping_pong.rs`, `#[cross_vm_contract]`) deploys the existing CosmWasm, EVM, and Solana ping-pong contracts unchanged and exposes one logical interface (`ping` / `receive_packet` / `acknowledge_packet` / `stats` / `port`). Each deployed handle carries an `on_after` hook that parses the emitted packet-lifecycle events (`SendPacket` / `ReceivePacket` / `WriteAcknowledgement` / `AcknowledgePacket`) into a shared ledger.
* `cross-vm-integration-tests`: a `Bridge` relayer (`tests/support/bridge.rs`) with a `BridgeLedger` (the append-only log of observed packets), a VM-agnostic `parse_packets(kind, &RawResponse)` parser (CosmWasm `wasm-`prefixed events, EVM `alloy` `SolEvent` decode, Solana base64 + borsh `emit!` decode), and a `port -> Endpoint` registry. Given `&mut MultiChainEnv`, `Bridge::relay` moves each recorded `SendPacket` to its destination contract (`receive_packet`) and the resulting `WriteAcknowledgement` back to the source (`acknowledge_packet`). The synchronous, read-only `on_after` hooks only record; the async relay is driven explicitly so it can reach `env` and the other chain. The test asserts the full packet lifecycle for every ordered VM pair.
* `cross-vm-integration-tests`: a `PingPongHarness` (`tests/harness/ping_pong.rs`) lifts the same relayer into the property-testing `Harness`, driven by the scenario, fuzz, invariant, and endurance runners. Operations are `Ping { src, dst }` and `Relay` (one *tick*: deliver only the packets pending at that moment, leaving the events it emits for the next tick), so a packet visibly steps `Send -> Receive/WriteAck -> Ack` across separate ticks. The `BridgeLedger` and `Bridge` live in the persisted `World`. Every transaction (user ping and the relay's own calls) is parsed by the same `parse_packets`; relay progress is derived from the recorded events (a `SendPacket` is pending until a `ReceivePacket` with its key appears; a `WriteAcknowledgement` until an `AcknowledgePacket`), not from side flags. No `on_after` hooks (every call goes through async `apply`, which records by parsing the returned `AppResponse`). Invariants at every step: `StatsMatchLedger` (on-chain `pings_sent` / `pongs_received` match the ledger's `SendPacket` / `AcknowledgePacket` counts per port) and `NoOrphanEvents` (no lifecycle event without its antecedent, so a bogus emission is caught). The endurance test drains all in-flight packets after the run and asserts the ledger fully settled. Gated behind the existing `fuzz` / `invariant` / `endurance` features; a scenario round-trip runs under default `cargo test`.
* `cross-vm-integration-tests`: new dev-dependencies `ping-pong` (path) and `base64`. Running the Solana arm needs `make compile-solana` to build `ping_pong.so`.

### Changed (harness: inject `(Ctx, World)` into a mode-typed runner, drop `Harness::setup`)

* `cross-vm-framework`: setup is no longer a `Harness` method. `Harness::setup` is removed; the trait now holds only `apply` / `op_kinds` / `generate_op` / `generate` / `invariants` / `check` / `advance` over the four associated types. Each test builds its own live `(Ctx, World)` (deploy, prime the model, establish op preconditions) and loads it into a runner with `r.setup(ctx, world)`. This lets every operation, combination, and endurance test set up differently instead of sharing one rigid `setup`.
* `cross-vm-framework`: the run shape is encoded in the runner's type via a phantom `RunMode` marker (`Fuzz` / `Invariant` / `Endurance` / `Scenario`), mirroring the `MultiChainEnv<Setup>` / `MultiChainEnv<Running>` typestate. `Runner<H, M>` is constructed as a shell (`Runner::fuzz` / `::invariant` / `::endurance` / `::scenario`, each taking `(harness, seed)`), then loaded with `setup(ctx, world)`. Type aliases `FuzzRunner<H>` / `InvariantRunner<H>` / `EnduranceRunner<H>` / `ScenarioRunner<H>` name each mode. Driver methods take `&mut self` and operate on the loaded env+world: `Fuzz`/`Invariant` share `run(ops, kinds, check_every)`; `Endurance` takes `run(EnduranceConfig)`; `Scenario` has `run_case` / `run_scenario` / `replay`.
* `cross-vm-framework`: `Mode`, `RunConfig`, `Runner::run`, `Runner::run_fuzz_operation`, `Runner::run_fuzz_combination`, and `Runner::run_fuzz_case` are removed. The per-case and per-run loops that lived inside the runner move out to the test layer: fan-out is now the `#[fuzz_runner]` macro's job (one libtest entry per case), and the persisted modes are single tests. `EnduranceConfig` (with a `new(duration)` builder plus `max_delay` / `check_every` / `advance_blocks`) replaces the endurance fields of `RunConfig`. `sub_seed(seed, case)` moves to `harness::rng` and is now public (re-exported from the prelude) so the generated tests and hand-written replays share one case derivation.
* `cross-vm-macros`: `fuzz_matrix!` is replaced by three attribute macros, `#[fuzz_runner]`, `#[invariant_runner]`, and `#[endurance_runner]`. Each takes `harness =`, `seed =` (and `cases =` for fuzz) and injects a seeded, mode-typed runner shell into the test fn's single `#[runner]`-marked argument (in the style of rstest's `#[case]`). The developer writes the whole body: their own setup, `r.setup(ctx, world)`, the `run(..)` call, and the asserts. `#[fuzz_runner]` emits one `#[tokio::test]` per case (`<name>_case_0` .. `<name>_case_<cases-1>`, seeded by `sub_seed(seed, i)`); the other two emit a single test.
* `cross-vm-macros` / `cross-vm-framework`: a negative seed in any runner macro (`seed = -1`) picks a fresh random seed per run via the new `random_seed()` helper (`harness::random_seed`, re-exported from the prelude) and prints it (for fuzz, one shared base across all cases), so a failure stays reproducible by copying the printed value back as a fixed `seed`. `seed` still defaults to 0.
* `cross-vm-framework`: a setup failure is no longer reported as a `RunReport { step: 0, FailureKind::Infra }`. Setup now runs in the test body before the runner drives anything, so a build failure surfaces as the setup function's `Err` (the caller `.expect()`s or `?`s it). The runner-mechanics `setup_failure_*` test and `Behavior::FailSetup` are updated/removed accordingly.
* Migrated the harness examples: the multi-chain counter and the DeFi vault lift their old `setup` bodies into free `*_setup(seed)` functions and rewrite each test to load the runner with `r.setup(ctx, world)`; the fuzz, invariant, and endurance tests use the new attribute macros, and the rstest matrices use `ScenarioRunner`.
* `cross-vm-integration-tests`: the heavier harness runs are gated behind cargo features `fuzz`, `invariant`, and `endurance` (one per mode), so the default `cargo test` runs only the scenario (rstest matrix) tests and the runner-mechanics self-tests. New Makefile targets `test-examples`, `test-harness`, `test-cross-vm`, `test-fuzz`, `test-invariant`, `test-endurance`, and `test-harness-all` cover the combinations.

### Changed (harness fuzz: per-op vs combination modes, shared generate_op)

* `cross-vm-framework`: fuzzing is split into two explicit modes and generation is decomposed. `Mode::Fuzz` and the `Harness::variants` isolation pass are removed. `Mode::FuzzOperation { op, runs }` runs `runs` randomized instances of a single operation kind, each on a fresh world (isolates one operation across many inputs). `Mode::FuzzCombination { kinds, cases, ops_per_case }` runs random mixed sequences, drawing from a restricted set of kinds or all of them (`kinds: None`). The old isolation pass was not actually fuzzing (it replayed fixed representative ops once each); per-op fuzz replaces it with randomized inputs.
* `cross-vm-framework`: the `Harness` trait gains an `OpKind` associated type (the data-free operation kinds), `op_kinds()`, and `generate_op(rng, world, kind)` (build a random op of a given kind). `generate` becomes a provided default (uniform kind choice then `generate_op`); override it only to bias the kind distribution, reusing `generate_op` for the data so generation logic lives in one place. `variants` is removed.
* `cross-vm-framework`: `Mode` and `RunConfig` are now generic over the operation kind (`Mode<K>` / `RunConfig<K>`, with `K = Harness::OpKind`) so the fuzz modes can name kinds. Existing `RunConfig::new(Mode::Invariant { .. })` call sites infer `K` from `Runner::run` and need no annotation. `Runner` gains `run_fuzz_operation` and `run_fuzz_combination`; `run_invariant` and each combination case now share the same `drive_random_case` core (single persisted case = one invariant run).

### Added (fuzz_matrix: one test per fuzz case)

* `cross-vm-framework`: `Runner::run_fuzz_case(seed, case, ops_per_case)` runs a single combination sequence (over all kinds) on a fresh world. The case is seeded by `sub_seed(seed, case)`, the same derivation `Mode::FuzzCombination` uses for its `case`-th sequence, so case `i` here is the exact sequence a full combination run would execute at index `i` (the canonical way to re-run or pin a single flagged case).
* `cross-vm-macros`: `fuzz_matrix! { name, harness, seed, cases, ops }` emits one `#[tokio::test]` per case index (`<name>_case_0` .. `<name>_case_<cases-1>`), each calling `run_fuzz_case`. This gives fuzz sequences the same per-case reporting, parallelism, and name-filtering that `#[rstest] #[values(..)]` gives a value grid. The case count is a compile-time choice (Rust's test harness fixes its case list before running); the tests are emitted at the call site so they share its imports (notably `Runner` from the prelude). Re-exported from the framework prelude.

### Changed (harness: MultiChainEnv as the core, World as persisted state)

* `cross-vm-framework`: the property-testing `Harness` trait no longer conflates the live system under test with bookkeeping. `state() -> World` is replaced by `setup() -> (Ctx, World)`. `Ctx` (new, `harness::Ctx`) wraps a started `MultiChainEnv<Running>` and is the live execution context threaded by `&mut` through `apply`, `check`, and `advance`. `World` now holds only persisted state (the in-memory shadow model, flags, and the addresses of contracts deployed or discovered so far); it carries no chains and no contract wrappers. A wrapper is rebuilt on demand from `Ctx` plus a stored address (`Ctx::chain(label)` hands out a cloned, state-sharing chain handle, then `Counter::instance(chain, addr)`). This is what makes a contract that creates another contract testable: `apply` records the child's address in `World` (read from the response events or a factory query) and a later step rebuilds a handle for it. `MultiChainEnv` gains `chain(label)` (cloned handle) and `advance_all(n)`; the `Harness::advance` default now warps every chain, so a multi-chain harness needs no override.
* `cross-vm-framework`: `Harness::check` returns the new `CheckOutcome` (`Held` / `Skipped(reason)` / `Violated(Violation)`) instead of `Result<(), Violation>`, so an invariant whose precondition has not occurred yet (for example "no counter incremented") declares itself not applicable rather than vacuously passing. `RunReport` gains a `skipped` count of how many checks were skipped over the run. `From<Result<(), Violation>>` is provided for `CheckOutcome` to ease migration.
* `cross-vm-macros`: the generated contract wrapper gains a public `address() -> Option<Account>` accessor (alongside the existing `instance(chain, address)`), so a caller can capture where a wrapper deployed and rebuild a fresh handle for the same instance.
* Migrated the harness examples to the new shape: the multi-chain counter and the DeFi vault build a `MultiChainEnv` in `setup()` and rebuild handles per step; the counter's `CountMatchesModel` invariant is `Skipped` until the first increment. The runner-mechanics test file (`harness/runner.rs`) is renamed to `harness/mechanics.rs` and its in-memory bank now runs against an empty `Ctx`, with an always-skipped invariant added to cover skip handling and the skip count.

### Changed (rename: TestingEnv -> MultiChainEnv)

It is written against the chain-agnostic layer: a `CounterSpec` trait with `#[cross_vm_contract(Counter)]` plus per-VM hooks (`cw_*` / `evm_*`), so `main` wraps each chain as an `AnyChain` and runs one identical `setup -> increment -> count` flow that the macro dispatches to the matching VM. A chain that fails (for example an unfunded wallet, surfaced by `setup`) is reported and does not stop the other; the run exits non-zero if any failed. Build the artifacts first (`make compile-solidity`, `make compile-cosmwasm`), then run `cargo run --manifest-path examples/scripts/Cargo.toml --bin deploy_counter`.

### Added (cross-VM dispatch macro)

* New `cross-vm-macros` crate (proc-macro) providing the `#[cross_vm_contract(StructName)]` attribute, re-exported through the framework prelude. It turns a trait of logical-method signatures into a cross-VM contract wrapper, generating the boilerplate the hand-written wrappers used to carry: the `StructName` struct (holding a `ContractBase`), the `new` / `instance` constructors, the `on_before` / `on_after` hook forwarders, and an `impl Spec for StructName` whose methods dispatch on the chain's VM to the matching `cw_*` / `evm_*` / `svm_*` hook. The developer writes only those per-VM hooks.
* Hooks are inferred, not annotated: a method whose declared Ok type is `AppResponse<_>` is wrapped in `run_before` / `run_after` (after-hooks need an execution response to observe), while a query like `count -> u64` is a plain dispatch. Return types in the spec trait are written as the bare Ok type (`AppResponse<()>`, `u64`, or omitted for `()`); the macro rewrites each to `Result<Ok, CrossVmError>`. Because the dispatchers are trait methods, call sites bring the spec trait into scope.
* The `support::Counter` test wrapper now uses the macro (a `CounterSpec` trait plus the per-VM hooks); the cross-VM and harness counter tests pass unchanged, proving generated dispatch matches the prior hand-written shape. Macro internals (codegen, hook inference, `&self` and generic-parameter guards) are covered by unit tests in the proc-macro crate.

### Added (Phase 8: wallet management)

* `cross-vm-core`: a VM-agnostic `WalletFactory`. Mnemonics are the only secret, so a `.env` holds nothing but BIP-39 phrases (one or more, each under its own variable). The wallet roster is the compile-time `WALLETS` const: each `WalletSpec` row names a label, a source (`WalletSource::Generate` for a fresh random mnemonic, or `WalletSource::Env(var)` to read a named `.env` variable), an account index, and an optional explicit HD path. `WalletFactory::from_env_file` loads the secrets (via `dotenvy`, without mutating the process env) and resolves every row; `from_secrets` does the same from an in-memory map (handy for tests); `in_memory` resolves only the `Generate` rows. Add a wallet by adding a const row, not a runtime call.
* `cross-vm-core`: a `WalletDeriver` trait (sibling of `ChainProvider`) that turns a mnemonic plus HD path into that ecosystem's signer. Each VM crate implements it on its chain handle with its own coin type and algorithm: EVM coin 60 via alloy `MnemonicBuilder` (`PrivateKeySigner`), Cosmos coin 118 via `bip39` seed plus cosmrs `bip32` derivation and the chain's bech32 prefix (`CosmosSigner`), Solana coin 501 via `bip39` seed plus SLIP-10 ed25519 (`SvmSigner`, an `Rc<Keypair>`).
* Per-wallet broadcast locks. The factory hands out a `tokio::sync::Mutex` owned guard per label; broadcast methods acquire it for the whole build, sign, broadcast sequence so the same wallet never sends two transactions at once (which would collide on the EVM nonce or Cosmos account sequence). An async mutex is required: a `std` mutex held across an `.await` would deadlock the current-thread runtime. Different wallets run in parallel; only same-wallet sends serialize.
* Broadcast methods now take a wallet label instead of a bare address: `EvmChain::deploy_create`/`call`, `CwChain::instantiate`/`execute_contract`, `SvmChain::send_transaction` accept a `&str` label, resolve it through the factory (deriving and caching the signer once), and lock for the broadcast. New `XxxChain::attach_wallets` shares the factory with a chain, and `wallet_address(label)` derives a wallet's address without locking (for funding or assertions in the setup phase).
* `cross-vm-framework`: `MultiChainEnv::<Setup>::with_env_file(path)` builds the factory from a `.env`, and `with_wallets(factory)` accepts a pre-built one; `start()` distributes it to every injected chain. `WalletFactory`, `WalletSource`, and `WALLETS` are re-exported in the prelude.
* EVM live-RPC write paths are now implemented: `EvmRpcProvider::deploy_create` and `call` sign with the derived `PrivateKeySigner` (alloy `EthereumWallet` plus recommended fillers) and broadcast, replacing the previous `Unimplemented`. Cosmos and Solana RPC broadcast remain `Unimplemented` for now: the wallet signer is plumbed through, but Cosmos `store_code` is blocked by `CwCode` being a cw-multi-test trait object (not wasm bytes), and the Cosmos/Solana broadcast return types (`AppResponse`, litesvm `TransactionMetadata`) are coupled to the mock backends; decoupling them is a focused follow-up.
* New deps: `dotenvy` and `bip39` (with `rand`) in the workspace; `alloy-signer-local` (mnemonic feature) for EVM; cosmrs `bip32` feature for Cosmos; `solana-keypair` (`seed-derivable`) plus `solana-derivation-path` for Solana; `tokio` (`sync`) promoted to a normal dependency where broadcasts run.
* New `.env.example`, a `wallet_quickstart` framework example (derive, fund, locked broadcast through the env), and tests: factory resolution and error cases, per-VM derivation determinism with a known-answer EVM vector, and per-wallet lock serialization. The cross-VM `counter` and `cross_vm` tests now drive every transaction through factory-derived wallets.

### Added (Phase 7: transaction hooks)

* `cross-vm-framework`: per-contract before/after transaction hooks on `ContractBase`. Register with `on_before` / `on_after`; a wrapper's method dispatcher fires them with `run_before` / `run_after` around the per-VM execution. An after-hook observes the uniform `AppResponse` (`label`, `kind`, `transaction_hash`, `gas_used`), so side-logic (indexer, bridge, event listener) reacts to a transaction without knowing the VM.
* Per-VM event accessors, exposed on `RawResponse` / `AppResponse` / `HookContext` (the shapes do not unify): `cosmwasm_events` (typed `Event`s), `evm_logs` (ABI `Log`s), `solana_logs` (program log lines; Anchor `emit!` events are base64 inside them). Each errors `WrongVm` off its VM. An after-hook watching all three matches on `kind()`.
* `cross-vm-solidity`: `EvmMockProvider::call` (and `EvmChain::call`) now returns `EvmExecution { output: Bytes, logs: Vec<Log> }` instead of bare `Bytes`. revm already reports the logs in its `ExecutionResult`; the provider previously discarded them. `static_call` (read path) still returns `Bytes`. New exports `EvmExecution` and a re-export of `alloy_primitives::Log`.
* Hooks are synchronous `FnMut`; async side-effects flow through a captured channel or `Rc<RefCell<_>>` buffer (the runtime is current-thread, futures are not `Send`). Both kinds return `Result<(), CrossVmError>`: a before-`Err` stops the transaction, an after-`Err` becomes the method's error. Fired in registration order; re-entry into the same contract's hooks panics on the `RefCell`.
* New exports `BeforeContext`, `HookContext`, `Hooks` (in the prelude). The VM-specific extraction (`transaction_hash`, `gas_used`, `cosmwasm_events`) moved onto `RawResponse`, with `AppResponse` delegating and a new `AppResponse::raw()` accessor; existing `AppResponse` accessors are unchanged.
* The `counter_cross_vm` example registers an after-hook and asserts it fires once per increment with VM-correct data across all three VMs; new unit tests cover before-abort, after-propagate, ordering, and the empty-registry no-op.
